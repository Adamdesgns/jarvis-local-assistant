# JARVIS Mobile v2 — Cameras, File Transfer, macOS Restyle

## Context

JARVIS Mobile shipped as a deliberate v1: **three screens** (pair, chat, offline),
24 lines of CSS, chat + press-and-hold voice only. It works, and Adam has it
paired on his iPhone over Tailscale HTTPS. Now that it's real, he wants it
finished properly:

- **He can't find the cameras** — correct, they were never built for mobile.
- **He wants to send files from the phone to the PC.**
- **He wants a macOS look** and to review every screen.

Decisions taken (2026-07-19): restyle the **phone app only** (desktop keeps its
amber Classic / cyan Command Center skins); cameras on phone = **snapshots +
alerts**, not live video; file transfer = **phone → PC only**.

This plan also clears the papercuts logged during the v1 build reviews — most
importantly the pairing QR still points at the old `http://` address, which is
now actively broken (iPhone refuses microphone access on non-HTTPS pages).

## Design direction: "macOS, but still JARVIS"

Neutral graphite surfaces with translucent frosted panels, generous rounded
rects (~12px), system font stack (`-apple-system, BlinkMacSystemFont, "SF Pro
Text", system-ui`), hairline borders, soft shadows, restrained motion — and the
**amber accent kept** so it still reads as JARVIS rather than a generic Apple
app. Honour `prefers-color-scheme` for light and dark, which is the macOS-native
behaviour and cheap to do in a stylesheet written from scratch.

The current `mobile.css` is 24 lines of ID-based rules with no reusable
components, so this is a **wholesale rewrite, not a refactor** — low risk (there
is no shared component library to break) but the design system has to be built
rather than extended.

## Architecture

### File upload (phone → PC)

No multipart parser exists anywhere in the repo and none is needed: `/api/voice`
already proves the pattern — **one file per POST, raw binary body, metadata in
headers**. Reuse it exactly.

- `POST /api/upload`, auth required. Headers: `Content-Type` (the file's type),
  `X-Filename`, `X-Destination` (an approved root path).
- Server validates: size cap (**25 MB**, matching `document-service.js:93`'s
  read cap — `readBody`'s current 10 MB default must be raised for this route
  only), filename sanitised, destination checked with the existing
  **`DocumentService.isAllowed(target)`** (`core/document-service.js:63-70`) —
  the same `path.resolve` + strict-prefix guard every desktop write already uses.
- New `DocumentService.createBinaryFile(location, name, buffer)` mirroring
  `createTextFile` (`document-service.js:222-232`): resolve location →
  `isAllowed` → `cleanName` → collision dedupe (`Name 2.ext`) → `writeFile` with
  the `wx` exclusive flag. A sibling method, **not** a change to `createTextFile`
  (which is string-only by design).
- `GET /api/folders` returns the approved roots so the phone can offer a
  destination picker.
- Phone: `<input type="file" multiple>` (iOS offers Photo Library / Files /
  Camera), destination `<select>`, per-file progress, clear success/failure per
  file. Multiple files upload sequentially, one POST each.

### Cameras (snapshots + alerts)

- `GET /api/cameras` → `cameras.listCameras()` (`core/camera/camera-service.js:233`).
- `GET /api/cameras/snapshot?key=…` → `cameras.getSnapshot(key, {manual:true})`
  (`camera-service.js:252`), returned as **raw `image/jpeg` bytes, not
  base64-in-JSON** — snapshots can be hundreds of KB and base64 through a
  synchronous `JSON.stringify` write would add real latency over cellular.
- `MobileServer` needs cameras injected. It is constructed before `cameras` in
  `main.js`, so pass a **lazy getter** (`getCameras: () => cameras`) — the same
  fix already used for the `look_at_camera` tool.
- Alerts: `cameras:alert` currently fans out only to Electron IPC. Wire it in
  `main.js` to also `mobileServer.pushEvent(deviceId, 'camera-alert', {...})`
  for every paired device — `pushEvent` and the `/api/events` SSE channel
  already exist and carry `agent-step`/`reply` today.
- **Explicitly out of scope: live video.** go2rtc binds loopback-only with no
  auth (`go2rtc-manager.js:44,48`), and Ring and Nest each bypass it with their
  own WebRTC paths — phone live view needs a reverse-proxy plus per-brand SDP
  relays. That is its own project.

### Papercuts to clear (from v1 review notes)

1. **Pairing QR points at `http://`** — now actively breaks the microphone. Add a
   `mobilePublicUrl` setting (default `''`), surface it in Settings → MOBILE, and
   have `mobile:pair` build the QR from it when set. Adam's value:
   `https://alienadam.taile7c34c.ts.net`
2. **App icon** — `apple-touch-icon` is an SVG, which iOS silently ignores (Adam
   already noticed the plain icon). Ship PNG icons (180×180 + 192/512 for the
   manifest).
3. **Non-401 HTTP errors render nothing** — a 500 clears the "Working…" strip and
   shows no bubble and no error. Surface a real message.
4. **A revoked key never surfaces** — `EventSource`'s `onerror` swallows the 401
   and retries silently forever. Detect and drop to the pairing screen.
5. **SSE `reply` racing the POST response** can suppress the spoken reply on a
   voice turn. Prefer the POST result for speech; treat SSE as the fallback.

  a renderer syntax error does not fail `npm test`, so this is the real net.

## Not in scope

Live camera video on the phone; PC → phone file download; a phone file browser;
tasks/memory/activity views on mobile; approving risky actions from the phone
(still desktop-only by design); the Siri Shortcut (separate, discussed).
